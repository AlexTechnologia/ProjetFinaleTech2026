📋 Synopsis et concept du projet (Jour 1)
Titre officiel
VeilCraft
Genre du projet
Jeu de survie 3D en vue à la première personne — FPS Survie Multijoueur
Synopsis
Le joueur se retrouve échoué sur une île mystérieuse entourée de brume. Le jour, il explore l'environnement procédural, récolte des ressources (bois, pierre, minerais) et craft des outils de plus en plus puissants. La nuit, des vagues d'ennemis débarquent sur l'île — gobelins, orcs et chevaliers des ténèbres — et le joueur doit survivre jusqu'à l'aube. Avec des amis, le tout devient encore plus intense : on peut jouer jusqu'à 4 en multijoueur P2P via WebRTC, sans avoir besoin d'un serveur dédié coûteux. L'île est générée de façon procédurale à partir d'une seed, donc chaque partie est différente.
Mécaniques principales
  • Récolte de ressources (bois, pierre, silex, minerai de fer, champignons)
  • Système de craft avec établis (Établi, Fournaise, Enclume, Chaudron, Table de flèches)
  • Gestion de jauges de vie, faim et endurance
  • Cycle jour/nuit avec vagues d'ennemis nocturnes
  • Multijoueur P2P via WebRTC + serveurs STUN/TURN
  • Génération procédurale de l'île basée sur une seed
  • Sauvegarde du monde dans le navigateur (localStorage)
  • Mode solo et mode multijoueur avec lobby et code de salle

Code JavaScript écrit / modifié player.js — Mouvement FPS avec gravité :
class Player {
  constructor(scene, camera) {
    this.health = 100; this.maxHealth = 100;
    this.hunger = 100; this.maxHunger = 100;
    this.stamina = 100; this.maxStamina = 100;
    this.velocity = new THREE.Vector3();
    this.onGround = true;
    this.inventory = new Array(30).fill(null);
  }
  update(dt) {
    // Appliquer gravité
    if (!this.onGround) this.velocity.y -= 20 * dt;
    // Décrémenter faim lentement
    this.hunger -= 0.5 * dt;
  }
}
world.js — Générateur seedé Mulberry32 :
function mulberry32(seed) {
  return function() {
    seed |= 0; seed = seed + 0x6D2B79F5 | 0;
    let t = Math.imul(seed ^ seed >>> 15, 1 | seed);
    t = t + Math.imul(t ^ t >>> 7, 61 | t) ^ t;
    return ((t ^ t >>> 14) >>> 0) / 4294967296;
  };
}

// Tous les clients avec la même seed = île identique !
Difficultés / bogues Le pointer lock sur Firefox ne fonctionnait pas pareil que Chrome — il fallait cliquer deux fois pour l'activer. Aussi, les collisions avec le terrain étaient bugguées : le joueur "tombait" à travers la surface si on se déplaçait trop vite (delta time trop grand).
Solutions / ressources consultées Pour le pointer lock, on a ajouté un event listener sur pointerlockchange pour détecter l'état. Pour les collisions rapides, on a plafonné le delta time à 0.05s max pour éviter que le joueur passe à travers. On a aussi ajouté un rayon vers le bas (raycast) pour détecter le sol précisément. Eric a regardé des solutions sur Stack Overflow.

Code JavaScript écrit / modifié entities.js — IA ennemis (pathfinding basique) :
class Enemy {
  update(dt, playerPos) {
    const dir = new THREE.Vector3()
      .subVectors(playerPos, this.mesh.position)
      .normalize();
    this.mesh.position.addScaledVector(dir, this.speed * dt);
    if (this.mesh.position.distanceTo(playerPos) < 1.5) {
      this.attack(dt); // Attaque toutes les 1.5s
    }
  }
}

// Gestionnaire de vagues nocturnes
class WaveManager {
  spawnWave(dayNumber) {
    const count = 3 + (dayNumber - 1) * 2;
    for (let i = 0; i < count; i++) this.spawnEnemy(dayNumber);
  }
}
main.js — Cycle jour/nuit :
function updateDayNight(dt) {
  dayTime += dt;
  const t = dayTime / DAY_DURATION; // 0=minuit, 0.5=midi
  const sunAngle = t * Math.PI * 2;
  sun.position.set(Math.cos(sunAngle)*50, Math.sin(sunAngle)*50, 0);
  renderer.setClearColor(skyColorForTime(t));
}
Difficultés / bogues Les ennemis se superposaient tous au même point quand ils mouraient et que leurs mesh restaient dans la scène. Aussi, la détection de clic pour récolter les ressources marchait parfois sur des objets derrière d'autres (le raycast ne filtrait pas bien).
Solutions / ressources consultées Pour les ennemis morts, on a ajouté un appel à scene.remove(enemy.mesh) et un nettoyage de la géométrie avec mesh.geometry.dispose() pour pas avoir de memory leaks. Pour le raycast, on a utilisé Raycaster.intersectObjects([]) avec recursive: true et on prend seulement le premier résultat. Doc Three.js Raycaster consultée.

Code JavaScript écrit / modifié audio.js — Son procédural avec Web Audio :
class AudioManager {
  playHitWood() {
    const osc = this.ctx.createOscillator();
    const gain = this.ctx.createGain();
    osc.type = 'sawtooth';
    osc.frequency.setValueAtTime(80, this.ctx.currentTime);
    osc.frequency.exponentialRampToValueAtTime(40,
      this.ctx.currentTime + 0.1);
    gain.gain.setValueAtTime(0.3, this.ctx.currentTime);
    gain.gain.exponentialRampToValueAtTime(0.001,
      this.ctx.currentTime + 0.15);
    osc.connect(gain); gain.connect(this.ctx.destination);
    osc.start(); osc.stop(this.ctx.currentTime + 0.2);
  }
}

Difficultés / bogues En multijoueur, les ressources détruites par Alex n'étaient pas supprimées sur l'écran d'Eric (désync). Aussi, deux joueurs qui frappent la même ressource en même temps causaient parfois des drops en double côté client. La latence sur le réseau de l'école était pas géniale (~80ms).
Solutions / ressources consultées Pour la sync des ressources : l'hôte est maintenant "autorité" — quand une ressource est détruite, l'hôte envoie un message RESOURCE_DESTROYED à tous les clients, et eux suppriment la ressource localement. Pour les drops en double, seul l'hôte génère les drops et les envoie via ITEM_DROPPED. Les clients reçoivent juste les positions des objets au sol. On a regardé des articles sur la "server authority" dans les jeux P2P.

Code JavaScript écrit / modifié save.js — Sauvegarde multislot :
class SaveManager {
  save(slot, worldData, playerData) {
    const saveObj = {
      timestamp: Date.now(),
      seed: worldData.seed,
      dayCount: worldData.dayCount,
      destroyedIds: worldData.destroyedIds,
      player: {
        pos: playerData.position,
        inventory: playerData.inventory,
        health: playerData.health
      }
    };
    localStorage.setItem(`veilcraft_save_${slot}`,
      JSON.stringify(saveObj));
  }
}
Corrections de bugs multiples : collision fix, UI sync, wave spawn fix.

Mais on a remarqué qu'en mode "rejoindre", si le code de salle était tapé en minuscule, ça ne trouvait pas la salle (l'hôte génère un code en majuscules). Un bug de casse ! Pour la molette macOS, on utilise event.deltaY > 0 plutôt que de se fier à la direction du scroll, ce qui est cohérent entre OS. On a testé sur Mac et Windows.

La partie du projet dont je suis le plus fier(ère) (code, design, mécanique...) :

La génération procédurale de l'île avec la seed — c'est ce qui fait que chaque partie est différente mais que tout le monde voit exactement la même île en multijoueur. Le fait d'utiliser Mulberry32 comme générateur pseudo-aléatoire déterministe, c'était une solution élégante qu'on a trouvé ensemble. L'IA des ennemis est aussi quelque chose dont je suis fier même si c'est basique — ils ont l'air de vraiment "pourchasser" le joueur.

Ce que j'aurais fait différemment ou amélioré avec plus de temps :

Je pense qu'on aurait dû tester le multijoueur plus tôt — au jour 3 ou 4 plutôt que jour 6. Les bugs de désynchronisation ont pris beaucoup de temps à régler. Aussi, j'aurais voulu implémenter un vrai pathfinding (A* ou même juste une grille de navigation) pour que les ennemis contournent les obstacles plutôt que de rentrer dedans.
